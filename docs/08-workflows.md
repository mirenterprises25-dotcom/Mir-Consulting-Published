# 08 · End-to-End Workflows

## 8.1 — Lead capture

```
Visitor fills /contact form
    ▼
POST /api/leads   (rate-limited 5/min/IP)
    ▼
Mongo: insert into  `leads`  (status=new)
    ▼
SMTP: notification email → COMPANY_EMAIL
    ▼
Admin sees the lead in /admin → Leads tab
    ▼
Admin can:  update status / add notes / delete / export CSV
```

**Failure modes**: SMTP down → still creates the lead in Mongo (email is best-effort).

## 8.2 — Invoice → email → payment

```
Admin → Invoices tab → "New invoice"
    Choose lead from dropdown (or type client details)
    Set currency, dates (shadcn Calendar), tax rate, line items
    Optional notes
    Save (draft)
    ▼
"Send invoice" button
    ▼
POST /api/admin/invoices/{id}/send
    ▼  (a) generate PDF via reportlab
       (b) Gmail SMTP → client email
       (c) flip status to `sent`
       (d) include public link → /invoice/{token}
    ▼
Client receives email → clicks "View online" → /invoice/{token}
    ▼
PublicInvoice.jsx renders invoice
       Buttons: [Download PDF] [Pay $X online]
    ▼
"Pay online" → POST /api/invoices/public/{token}/checkout
    ▼ returns { url } → redirect to Stripe Checkout
Stripe Checkout
    ▼ on success → /invoice/{token}?payment=success&session_id=cs_…
PublicInvoice polls GET /api/invoices/public/{token}/checkout/{session_id}
    every 2s for up to 60s.
    First time `payment_status=="paid"` → backend flips invoice → status=paid,
    paid_at=now, and updates payment_transactions row.
    Toast: "Payment received. Thank you!"
```

**Idempotency**: Both the polling endpoint and the Stripe webhook check the
`payment_transactions.payment_status` before flipping. Re-running either is safe.

## 8.3 — CMS publishing (Insights, Case Studies, Videos)

```
Admin → Insights / Case Studies tab → "New <type>"
    Fill title (auto-generates slug), excerpt, content (markdown), cover image upload
    "AI translate to DE/ES" if needed → form gets replaced with translation
        → change slug + save as a new article
    Click "Save draft" or "Publish"
    ▼
POST /api/admin/posts (or /case-studies)
    ▼  - markdown content stored as-is
       - cover_image stored as /api/media/blog/<file>
       - status: draft (hidden) or published (visible)
    ▼
Public list endpoint: GET /api/posts (only `status=published`)
    Public hub: GET /api/work → returns merged Insights + Case Studies + Videos
       used by /our-work page, with type + category filters.
```

## 8.4 — Media upload

```
Admin clicks "Upload" in any MediaUpload widget
    Select file
    ▼
POST /api/admin/media/upload  (multipart, with auth)
    ▼ backend slugifies filename + adds UUID
    ▼ httpx PUT to GitHub Contents API
    ▼ on success: returns { path, url }
URL format: /api/media/<folder>/<filename>
    ▼ frontend stores that URL in the resource
When the public site later renders the image:
    GET /api/media/<folder>/<filename>
    ▼ backend fetches the raw file from GitHub (cached LRU)
    ▼ returns bytes with Content-Type from extension + Cache-Control: public, max-age=86400
```

**Fallback**: if upload fails (e.g., GitHub PAT issue), the admin can still paste
an absolute URL (Cloudinary, S3, etc.) into the same field — the form accepts any URL.

## 8.5 — Admin password reset (magic link)

```
Admin clicks "Forgot password?" on /admin
    Enters their email
    ▼
POST /api/admin/forgot-password
    ▼ writes admins.reset_token + reset_token_expires_at (30min)
    ▼ Gmail SMTP → admin email with link to /admin/reset/<token>
Admin opens link in any browser
    ▼
POST /api/admin/validate-reset-token   (frontend pre-check)
    Admin enters new password (twice)
    ▼
POST /api/admin/reset-password
    ▼ bcrypt-rehashes the password
    ▼ wipes reset_token + reset_token_expires_at
Toast: "Password reset. You can now sign in."
```

## 8.6 — Multi-language site (i18next)

- Visitor's preferred language is detected from `navigator.language`, fallback `en`.
- Toggle in the navbar persists choice to `localStorage`.
- All UI strings live in `/app/frontend/src/locales/{en,de,es}/<namespace>.json`.
- CMS articles are NOT auto-translated at runtime — each translated article is its own row in the DB. The admin uses the AI translate button to produce a draft.
- SEO hreflang tags are generated by `Seo.jsx`: every page emits `<link rel="alternate" hreflang="en|de|es|x-default">` pointing at the same URL on the production canonical (`REACT_APP_SITE_URL`).
